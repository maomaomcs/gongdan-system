## 摘要

随着中小学校园信息化程度不断提高，校内电脑、打印机、一体机、交换机等IT设备数量迅速增长，设备故障报修与资产管理成为学校后勤信息化的两大痛点。传统模式下，师生报修依赖电话或口头传达，后勤人员难以跟踪处理进度；设备台账多以零散Excel维护，信息易失真、难统计。针对上述问题，本文设计并实现了一套面向中小学的校园设备报障工单与IT资产管理系统。

系统采用前后端分离架构，后端基于 Spring Boot 3 与 Spring Data JPA 构建 RESTful 接口，使用 MySQL 存储数据，通过 BCrypt 加密口令、令牌落库实现认证与授权；前端基于 Vue 3、Vite 与 Element Plus 构建响应式单页应用，并支持 PWA 安装。系统实现了报修工单全生命周期管理、处理进度跟踪、图片上传与压缩、钉钉群消息推送与快捷操作、数据统计可视化、IT资产台账管理、Excel 批量导入导出、保修到期预警以及"设备—工单"关联等功能。系统已部署于云服务器稳定运行，并通过功能测试与部署验证。

测试结果表明，系统功能完整、运行稳定，能够有效提升校园报修处理效率与资产管理规范化水平，具有较强的实用价值。

**关键词**：报修工单；IT资产管理；Spring Boot；Vue；前后端分离

## Abstract

With the continuous improvement of campus informatization in primary and secondary schools, the number of IT devices such as computers, printers, all-in-one machines and switches has grown rapidly. Device fault reporting and asset management have become two major pain points of school logistics informatization. In the traditional mode, fault reporting relies on phone calls or verbal communication, making it difficult for logistics staff to track progress; asset ledgers are usually maintained in scattered Excel files, which are prone to distortion and hard to analyze. To address these problems, this thesis designs and implements a campus device fault-reporting ticket system integrated with IT asset management for primary and secondary schools.

The system adopts a front-end and back-end separated architecture. The back end is built with Spring Boot 3 and Spring Data JPA to provide RESTful APIs, uses MySQL for data storage, and implements authentication and authorization through BCrypt password hashing and persistent tokens. The front end is a responsive single-page application built with Vue 3, Vite and Element Plus, supporting PWA installation. The system implements the full lifecycle management of repair tickets, progress tracking, image upload and compression, DingTalk group message push with quick actions, data statistics visualization, IT asset ledger management, Excel batch import/export, warranty expiration warning, and device-ticket association. The system has been deployed on a cloud server and verified through functional testing.

The test results show that the system is complete in function and stable in operation, which can effectively improve the efficiency of campus fault handling and the standardization of asset management, and has strong practical value.

**Keywords**: Repair Ticket; IT Asset Management; Spring Boot; Vue; Front-end and Back-end Separation

## 第1章 绪论

### 1.1 研究背景与意义

近年来，中小学校园信息化建设快速推进，多媒体教室、录播教室、机房、办公终端等大量投入使用。以本系统的应用单位——成都市石室联中132学校为例，全校拥有台式电脑、笔记本、打印机、一体机、交换机、投影仪等各类IT设备数百台，分布于教学楼、办公楼、机房等多个区域。这些设备一旦发生故障，将直接影响正常教学秩序。

然而，学校后勤信息化水平相对滞后，主要体现在两个方面：

（1）**报修流程不规范**。师生报修多依赖电话、微信或当面口头告知，缺乏统一入口。报修信息容易遗漏，后勤人员难以排序处理，也无法向报修人反馈进度，常出现"报了没人管""不知道修没修好"的情况；管理者也缺乏数据依据评估维修工作量与响应时效。

（2）**资产台账难维护**。设备台账多由个人用零散的Excel文件维护，字段不统一、更新不及时，设备的型号、IP、保修期、位置、责任人等信息难以准确掌握，更无法与报修记录关联，导致"哪台设备老坏""哪些设备快过保"等问题无从分析。

针对上述痛点，本文设计并实现一套集**报障工单**与**IT资产管理**于一体的校园信息化系统，将报修流程线上化、标准化，将资产台账集中化、结构化，并通过"设备—工单"关联打通两个模块。系统的研究与实现具有以下意义：

- **实用意义**：为学校提供一套开箱即用、成本低廉（开源技术栈、单机部署）的后勤信息化工具，切实提升报修处理效率与资产管理规范化水平。
- **技术意义**：完整实践了前后端分离、RESTful API、ORM、令牌认证、消息推送、数据可视化、PWA 等主流Web开发技术，具有一定的综合性与工程性。

### 1.2 国内外研究现状

在IT服务管理（ITSM）与IT资产管理（ITAM）领域，国内外已有较为成熟的开源与商业系统：

- **GLPI**：法国开源的IT资产与服务台管理系统，功能全面，支持资产管理、工单、知识库，并可配合 FusionInventory/GLPI-Agent 自动扫描网络设备。功能强大但部署与配置相对复杂，界面偏专业，对中小学校后勤人员不够友好。
- **Snipe-IT**：开源资产管理系统，界面友好、支持扫码借还，但侧重资产，报修工单能力较弱。
- **禅道、Zendesk、ServiceNow** 等：面向企业IT或客服场景，功能重、成本高，与中小学校园的轻量化需求不匹配。

上述系统大多面向企业级通用场景，存在功能冗余、部署复杂、本地化程度不足、缺乏针对校园场景（如按教室报修、钉钉通知、移动端扫码报修）的定制等问题。因此，面向中小学校园实际需求、轻量易用、可自主可控的定制化系统仍有较大价值，这也是本文的出发点。

### 1.3 主要研究内容

本文的主要工作包括：

1. 调研中小学校园报修与资产管理的实际业务流程，完成系统需求分析；

4. 实现报修工单与资产设备的关联，支持按设备查询历史报障记录；
5. 完成系统的功能测试与云服务器部署验证。

## 第2章 相关技术介绍

本章介绍系统实现所依赖的主要技术。撰写时可对每种技术展开2~4段，说明其原理、特点及在本系统中的作用。

### 2.1 后端技术

**Spring Boot 3**：基于 Spring 框架的快速开发脚手架，采用"约定优于配置"的理念，内置 Tomcat 容器，可将应用打包为单个可执行 Jar，简化部署。本系统后端基于 Spring Boot 3.4（要求 Java 17）构建，通过起步依赖快速集成 Web、数据访问、校验等能力。

**Spring MVC 与 RESTful API**：使用 `@RestController`、`@RequestMapping` 等注解定义 REST 接口，以 JSON 作为前后端数据交换格式，接口按资源与动作划分（如 `/api/admin/assets`）。

**Spring Data JPA 与 Hibernate**：JPA 是 Java 持久化规范，Hibernate 为其实现。通过实体类与注解映射数据库表，`JpaRepository` 提供通用增删改查，`JpaSpecificationExecutor` 支持动态条件查询（本系统的多条件分页筛选即基于 Specification 实现），`ddl-auto=update` 可根据实体自动维护表结构。

**BCrypt**：一种自带盐值、计算开销可调的口令哈希算法，用于安全存储用户口令。系统使用 spring-security-crypto 的 `BCryptPasswordEncoder` 对管理员与教师口令进行加密。

**Apache POI**：用于读写 Microsoft Office 文档的 Java 库。本系统使用其 XSSF 组件生成 `.xlsx` 报表（工单导出、资产导出、导入模板）并解析上传的 Excel（资产批量导入）。

### 2.2 前端技术

**Vue 3**：渐进式 JavaScript 框架，采用组合式 API（Composition API）与响应式数据绑定，组件化开发，适合构建单页应用（SPA）。

**Vite**：新一代前端构建工具，开发期基于原生 ESM 提供极速热更新，生产期基于 Rollup 打包。

**Element Plus**：面向 Vue 3 的桌面端组件库，提供表格、表单、弹窗、抽屉、时间线等丰富组件，本系统 UI 主要基于其构建，并通过覆盖主题变量实现"石室书院风"配色。

**Vue Router / Axios / ECharts**：分别用于前端路由（history 模式）、HTTP 请求封装（拦截器自动附加令牌）、统计图表绘制。

**PWA（渐进式 Web 应用）**：通过 manifest 与 Service Worker，使 Web 应用可"安装到桌面"、具备类原生体验，适合移动端师生快捷报修。

### 2.3 数据库

**MySQL**：主流开源关系型数据库。本系统开发环境使用 MySQL 8.0，生产环境使用 5.7，字符集 utf8mb4。数据表由 JPA 根据实体自动创建与维护。

### 2.4 部署与第三方能力

**云服务器与 systemd**：系统部署于腾讯云 CentOS 服务器，以 systemd 服务方式管理进程，实现开机自启与故障重启。

**钉钉群机器人**：通过钉钉自定义机器人 Webhook 推送新工单、催单、取消等消息，并利用 ActionCard 在群内提供"认领/已解决/取消"快捷按钮，按钮请求经服务端 HMAC 签名校验防篡改。

## 第3章 系统需求分析

### 3.1 可行性分析

**技术可行性**：系统采用 Spring Boot、Vue、MySQL 等成熟、开源、文档完善的主流技术，社区活跃、学习资料丰富，开发工具链免费，技术上完全可行。

**经济可行性**：全部采用开源软件，无授权费用；可部署于一台低配云服务器（如 2 核 2G）或校内服务器，运行成本低廉，学校易于承担。

**操作可行性**：系统面向师生与后勤人员，报修端界面简洁、移动优先、支持 PWA 安装；管理端参照常见后台习惯设计，并做移动端适配，普通用户无需培训即可上手。

### 3.3 功能需求分析

按模块划分，系统功能需求如下（撰写时建议为每个用例配用例图，见 3.5）。

**（1）用户与权限模块**
- 教师注册（需填写管理员发放的邀请码）、登录、登出；
- 管理员登录、修改口令、管理管理员账号（新增、停用、删除、改密）；
- 管理员管理教师账号（启停、重置口令、设为管理员、删除）；
- 注册邀请码管理（生成、设置备注与使用次数上限、启停、删除）。

**（2）报修工单模块**
- 教师提交报修：填写位置、故障类型、问题简述、详细描述、紧急程度，可选填设备编号、上传现场照片（最多6张）；
- 工单状态流转：待处理 → 处理中 → 已解决 → 已关闭，另有终态"已取消"；
- 教师端：查看"我的报修"（分页、按状态/关键词筛选）、催单（限频）、取消（仅待处理时）、进展未读红点提醒；
- 管理端：工单列表（多条件筛选、分页、最新置顶、超时标红）、详情、更新状态与处理结果、添加跟进记录；
- 游客：凭工单号查询处理进度。

**（3）通知模块**
- 新工单、催单、取消等事件通过钉钉群机器人推送；
- 钉钉群内可通过卡片按钮直接将工单置为"处理中/已解决/已取消"。

**（4）统计与导出模块**
- 按状态、故障类型、位置、月份等维度统计并以图表展示；
- 统计平均解决时长、未完成工单数等指标；
- 工单按筛选条件导出为 Excel。

**（5）IT资产管理模块（CMDB）**
- 资产台账的增、删、改、查（分页、按类型/状态/位置/关键词筛选）；
- 资产字段：资产编号、类型、品牌型号、序列号、IP、MAC、位置、责任人、使用科室、购入日期、保修到期、供应商、采购单号、状态、备注；
- 保修到期自动计算并预警（有效/即将到期/已过保）；
- 资产台账导出 Excel；下载导入模板、按模板批量导入（编号唯一校验、重复跳过、返回导入结果明细）；
- 资产统计（按类型、状态分布）。

**（6）设备—工单关联模块**
- 报修时填写设备编号，即可将工单与资产台账中的设备关联；
- 在资产台账中点击某设备，查看其全部历史报障记录（报障次数、状态、处理结果）。

**（7）系统设置模块**
- 配置钉钉机器人；
- 配置报修选项（故障类型、常用位置、常见问题FAQ、超时小时数）。

## 第4章 系统设计

### 4.1 系统总体架构设计

系统采用**前后端分离**的三层架构，整体分为表现层、应用层与数据层：

```
┌─────────────────────────────────────────────┐
│  表现层（前端 SPA，Vue3 + Element Plus + PWA） │
│  教师端(报修/我的报修)   管理端(工单/资产/统计/设置) │
└───────────────┬─────────────────────────────┘
                │ HTTP + JSON (RESTful, 携带令牌)
┌───────────────▼─────────────────────────────┐
│  应用层（后端 Spring Boot）                    │
│  Controller → Service → Repository            │
│  拦截器鉴权 · 令牌存储 · 全局异常 · 定时任务      │
│  钉钉推送 · Excel导入导出 · 图片存储压缩          │
└───────────────┬─────────────────────────────┘
                │ JPA / JDBC
┌───────────────▼─────────────────────────────┐
│  数据层（MySQL）+ 本地文件存储（上传图片）        │
└─────────────────────────────────────────────┘
                │ 外部：钉钉群机器人 API
```

生产部署上，前端经 Vite 打包后的静态资源由后端一并托管（打入同一 Jar 的 `static` 目录），因此线上仅运行一个 Spring Boot 服务（端口 8082），由 systemd 守护，降低了部署与运维复杂度。

### 4.3 后端分层设计

后端遵循经典分层，包结构（`com.school.ticket`）如下：

- `entity`：JPA 实体，映射数据库表；
- `repository`：数据访问接口，继承 `JpaRepository`；
- `service`：业务逻辑，事务边界；
- `controller`：REST 接口，参数校验与响应封装；
- `dto`：请求/响应数据传输对象（多用 Java record）；
- `web`：拦截器、令牌存储、全局异常处理、Web 配置；
- `config`：配置属性、初始化、定时任务、安全 Bean。

请求处理链路为：`Controller` 接收并校验请求 → 调用 `Service` 处理业务与事务 → 通过 `Repository` 读写数据库 → 组装 `DTO` 返回。跨切面能力（鉴权、异常、CORS、静态回退）在 `web` 层统一处理。

### 4.6 安全设计

- **认证**：登录成功后由服务端签发令牌并落库（auth_token 表），前端存于 localStorage，后续请求经 `Authorization: Bearer` 头携带；令牌设有效期（默认7天），定时任务清理过期令牌。
- **授权**：`AdminAuthInterceptor` 拦截 `/api/admin/**` 要求 ADMIN 角色，`UserAuthInterceptor` 拦截 `/api/user/**` 要求 USER/ADMIN，登录注册接口放行。
- **口令安全**：管理员与教师口令均以 BCrypt 加盐哈希存储，不可逆。
- **注册管控**：教师注册需管理员发放的邀请码，防止任意注册。
- **文件安全**：上传图片仅保留纯文件名并校验存在性，防目录穿越；图片上传即压缩，定时清理过期文件。
- **钉钉回调安全**：群按钮请求经服务端密钥 HMAC 签名校验，防止 URL 被篡改伪造。

api.interceptors.request.use((config) => {
  const url = config.url || ''
  if (url.startsWith('/admin')) {
    const t = localStorage.getItem('admin_token')
    if (t) config.headers.Authorization = 'Bearer ' + t
  } else if (url.startsWith('/user') && !url.includes('/login') && !url.includes('/register')) {

  }

### 5.3 报修工单全流程实现

**工单号生成**：以 `BX + yyyyMMdd + 4位十六进制随机数` 生成，并校验唯一性，兜底追加时间戳。

**状态流转与约束**：更新工单时校验状态合法性；"已取消"为报修人终态，锁定不可再改；状态或处理结果变化时置报修人未读标记，用于"我的报修"红点提醒。核心更新逻辑（节选）：

```java
@Transactional
public TicketResponse update(Long id, UpdateTicketRequest req) {
    Ticket t = ticketRepo.findById(id).orElseThrow(() -> new ApiException(404, "工单不存在"));
    if ("已取消".equals(t.getStatus()))
        throw new ApiException(400, "该工单已被报修人取消,不能再修改");
    boolean progress = false;
    if (req.status() != null) {
        if (!props.getStatuses().contains(req.status())) throw new ApiException(400, "非法状态");
        if (!req.status().equals(t.getStatus())) progress = true;
        t.setStatus(req.status());
        if ("已解决".equals(req.status()) && t.getResolvedAt() == null) t.setResolvedAt(LocalDateTime.now());
    }
    if (req.assetNo() != null) t.setAssetNo(trimOrNull(req.assetNo()));
    // ... handler / resolution
    if (progress) t.setUserUnread(true);
    ticketRepo.save(t);
    return withLogs(t);
}
```

**多条件分页查询**：基于 `JpaSpecificationExecutor` 动态拼接查询条件，支持按状态、类型、位置、关键词过滤并分页，关键词可匹配标题、描述、报修人、位置、工单号、设备编号。

### 5.4 IT资产管理与批量导入实现

**动态筛选与保修预警**：资产列表同样基于 Specification 实现多条件分页；响应对象根据保修到期日期计算剩余天数与保修状态（有效/即将到期/已过保，阈值30天），前端据此对临期、过保设备做颜色高亮。

**Excel 批量导入**：使用 Apache POI 解析上传的 `.xlsx`，逐行读取并校验，资产编号为空或类型为空的行记为失败，编号已存在的行跳过，其余入库，最终返回导入结果明细。核心解析逻辑（节选）：

```java
try (Workbook wb = new XSSFWorkbook(file.getInputStream())) {
    Sheet sheet = wb.getSheetAt(0);
    for (int r = 1; r <= sheet.getLastRowNum(); r++) {  // 跳过表头
        Row row = sheet.getRow(r);
        if (row == null || isEmptyRow(row, fmt)) continue;
        total++;
        String assetNo = cellStr(row, 0, fmt), type = cellStr(row, 1, fmt);
        if (assetNo == null) { failed++; errors.add("第"+(r+1)+"行:资产编号为空"); continue; }
        if (assetRepo.existsByAssetNo(assetNo)) { skipped++; errors.add("第"+(r+1)+"行:编号已存在"); continue; }
        Asset a = new Asset();
        a.setAssetNo(assetNo); a.setType(type);
        a.setPurchaseDate(cellDate(row, 10, fmt));
        a.setWarrantyEnd(cellDate(row, 11, fmt));
        // ... 其余字段
        assetRepo.save(a); success++;
    }
}
return new AssetImportResult(total, success, skipped, failed, errors);
```

其中日期单元格兼容 Excel 原生日期格式与文本格式（`yyyy-M-d`），解析失败则置空而不使整行失败，提升导入容错性。

## 第7章 总结与展望

### 7.1 工作总结

本文针对中小学校园设备报修与IT资产管理的实际痛点，设计并实现了一套集报障工单与资产管理于一体的信息化系统。主要完成的工作包括：

2. 基于 Spring Boot + Vue 前后端分离架构，设计了系统总体架构、功能模块、数据库与接口；
3. 实现了报修工单全生命周期管理、图片上传压缩、钉钉通知与群快捷操作、数据统计可视化、Excel 导入导出、IT资产台账管理、保修预警以及设备—工单关联等功能；

### 7.2 不足与展望

系统仍有可改进之处，后续可从以下方面完善：

3. **自动化采集**：可结合网络扫描（如 SNMP/Agent）自动发现并采集设备信息，减少手工录入；
4. **数据备份**：增加数据库与上传文件的定期自动备份；
5. **移动端深化**：可进一步开发微信小程序或封装 App，提升移动端体验；
6. **满意度评价**：工单解决后增加报修人满意度评价与重开工单机制。

## 附录A 核心数据表建表语句（真实导出）

> 以下 DDL 由系统运行后从 MySQL 真实导出（`mysqldump --no-data`），由 JPA 依据实体自动生成。完整脚本见 `docs/schema.sql`。

```sql
-- 报修工单表
CREATE TABLE `ticket` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `code` varchar(32) NOT NULL,
  `reporter` varchar(64) NOT NULL,
  `contact` varchar(64) DEFAULT NULL,
  `location` varchar(128) NOT NULL,
  `asset_no` varchar(64) DEFAULT NULL,
  `category` varchar(32) NOT NULL,
  `title` varchar(128) NOT NULL,
  `description` varchar(1000) DEFAULT NULL,
  `urgency` varchar(8) NOT NULL,
  `status` varchar(8) NOT NULL,
  `handler` varchar(64) DEFAULT NULL,
  `resolution` varchar(1000) DEFAULT NULL,
  `images` varchar(1000) DEFAULT NULL,
  `user_id` bigint DEFAULT NULL,
  `urge_count` int NOT NULL,
  `last_urged_at` datetime(6) DEFAULT NULL,
  `user_unread` bit(1) NOT NULL,
  `created_at` datetime(6) DEFAULT NULL,
  `updated_at` datetime(6) DEFAULT NULL,
  `resolved_at` datetime(6) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_code` (`code`),
  KEY `idx_status` (`status`),
  KEY `idx_category` (`category`),
  KEY `idx_location` (`location`),
  KEY `idx_ticket_asset_no` (`asset_no`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- IT资产台账表
CREATE TABLE `asset` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `asset_no` varchar(64) NOT NULL,
  `type` varchar(32) NOT NULL,
  `brand_model` varchar(128) DEFAULT NULL,
  `serial_no` varchar(128) DEFAULT NULL,
  `ip` varchar(64) DEFAULT NULL,
  `mac` varchar(64) DEFAULT NULL,
  `location` varchar(128) DEFAULT NULL,
  `owner` varchar(64) DEFAULT NULL,
  `department` varchar(64) DEFAULT NULL,
  `purchase_date` date DEFAULT NULL,
  `warranty_end` date DEFAULT NULL,
  `supplier` varchar(128) DEFAULT NULL,
  `purchase_order` varchar(64) DEFAULT NULL,
  `status` varchar(16) NOT NULL,
  `remark` varchar(1000) DEFAULT NULL,
  `created_at` datetime(6) DEFAULT NULL,
  `updated_at` datetime(6) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_asset_no` (`asset_no`),
  KEY `idx_asset_type` (`type`),
  KEY `idx_asset_status` (`status`),
  KEY `idx_asset_location` (`location`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

（其余 admin_user、app_user、auth_token、invite_code、ticket_log、system_setting 六张表的完整 DDL 见 `docs/schema.sql`。）
